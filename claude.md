# 역할 및 목표
당신은 Rust 시스템 엔지니어이자 경량 블록체인 아키텍트입니다. 저사양 홈서버(싱글보드 컴퓨터 또는 저전력 미니 PC)에서 메모리와 CPU 오버헤드 없이 데몬으로 상주하며 팽팽 돌 수 있는 'Pure Rust 기반의 초경량 블록체인 코어' 아키텍처와 구현 코드를 작성해 주세요.

# 개발 환경 및 제약 조건
1. 언어 및 런타임: Rust (최신 Stable 버전), 비동기 처리를 위해 `tokio` 사용.
2. 스토리지: 외부 독립형 DB(PostgreSQL, Redis 등)를 절대 쓰지 않고, 프로세스 내에 임베디드되는 고성능 KV 스토리지인 `sled`를 사용해 상태(State)와 블록 데이터를 영속화합니다.
3. 합의 알고리즘 및 네트워크: 복잡한 PoW/PoS나 거대한 P2P 네트워킹 레이어는 배제합니다. 홈서버 단독 구동 또는 신뢰할 수 있는 소수의 노드만 트랜잭션을 제출하는 '단일 시퀀서(Single Sequencer) / PoA(권한 증명)' 구조로 단순화하여 리소스 소모를 최소화합니다.
4. 트랜잭션 페이로드: 단순 토큰 송금이 아닌, 임의의 데이터(예: 파일 해시값, 메타데이터 스트링)를 담을 수 있는 범용적인 구조로 설계해 주세요.

# 핵심 구현 요구사항

1. Cargo.toml 설정: 
   - `tokio` (full features), `sled`, `serde` (derive), `bincode` (직렬화), `sha2` 또는 `blake3` (해싱) 등 필수 디펜던시만 포함.

2. 데이터 모델 (Data Structures):
   - `Transaction`: id(해시), payload(String 또는 Vec<u8>), timestamp, signature(선택사항, 구조만 명시).
   - `Block`: header(index, prev_block_hash, merkle_root, timestamp), transactions(Vec<Transaction>), hash(블록 해시).
   - `BlockchainState`: 현재 최신 블록 높이(Height)와 커밋된 데이터들의 상태를 관리하는 구조체.

3. 스토리지 레이어 (`sled` 연동):
   - 블록 데이터 저장용 트리와 상태 데이터(World State) 저장용 트리를 `sled::Db` 내에서 분리하여 관리.
   - 키-값 저장 시 `bincode`를 활용해 효율적으로 직렬화/역직렬화하는 로직 구현.
   - 서버가 재시작되어도 기존 `sled` 디렉토리에서 마지막 블록을 읽어와 체인을 그대로 이어갈 수 있는 초기화(Initialization) 로직 포함.

4. 코어 로직:
   - 트랜잭션 풀(MemPool) 관리 (간단한 메모리 내 큐).
   - 주기적으로 또는 트랜잭션이 쌓였을 때 새 블록을 생성(Mining/Minting)하고 `sled`에 커밋하는 비동기 루프 루틴.
   - 블록 생성 시 이전 블록 해시 검증 및 데이터 무결성 검증 로직.

5. 메인 함수 (`main.rs`):
   - `tokio::main` 환경에서 스토리지 초기화 -> 가상의 트랜잭션 몇 개를 생성하여 풀에 삽입 -> 비동기로 블록이 안전하게 생성되고 `sled` DB에 영속화되는 흐름을 보여주는 완전하고 실행 가능한 단일 파일 예제 코드로 작성해 주세요.
